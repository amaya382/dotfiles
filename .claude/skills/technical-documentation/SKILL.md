# Technical Documentation Skill

## Principles

- パラグラフライティング: 1パラグラフ(セクション)1トピック。冒頭文でトピックを述べ、後続文で裏付ける
- 簡潔さ: 同じ趣旨の文を繰り返さない。主張とその論拠以外の余計な修飾を書かない。短く書けるなら短く書く
- 構造化: 「情報の階層化」「情報の並列化」「情報の比較」「時間軸」「因果関係」の観点で情報を整理する。文章だけでなく箇条書き・表・図を活用する
- 論理的な章構成: セクション間に明確な流れを持たせ、読者が迷わない順序で情報を配置する
- 強調: 強調すべき事柄のみに太字を使う

## Constraints

- 曖昧な推測に基づく文を書かないこと。不足情報は自主的に調べ、それでも不明な場合は AskUserQuestion でユーザーに確認する
- カレントプロジェクトに `.claude/rules/` 等のドキュメント規約がある場合は遵守すること

## Mode 判定

ユーザーの入力から以下のいずれかのモードを判定する。判断に迷う場合は AskUserQuestion で確認する。

| モード | 条件 | フロー |
|--------|------|--------|
| 新規執筆 | ドキュメントがまだ存在しない | Bootstrap phase → Polishing phase |
| 改訂 | 既存ドキュメントの特定箇所を修正・加筆したい | 対象セクション特定 → 修正 → Polishing phase (対象部分のみ) |
| 推敲 | 既存ドキュメント全体の品質を改善したい | Polishing phase のみ |

## Bootstrap phase (新規執筆モード)

各ステップが確定するまで次のステップに進まないこと。コンテキストの入力は任意のタイミングで受け付ける。

### Step 0: プロジェクトコンテキスト確認

- カレントプロジェクトに `.claude/rules/` 配下のドキュメント規約ファイルがあるか確認する
  - ある場合: 規約の命名規則・frontmatter 要件・配置ルール・図表記法を以降のステップで遵守する
- カレントプロジェクトに専用のドキュメント作成スキルがあるか確認する
  - ユーザーのリクエストがその専用スキルの対象範囲に該当する場合: 専用スキルの使用を AskUserQuestion で提案する

### Step 1: テーマ・目的・想定読者の確定 (Plan mode で実行)

以下を AskUserQuestion でユーザーに確認する:

1. 誰に (想定読者): 技術レベル、前提知識
2. 何を (テーマ・目的): 伝えたい情報とその目的
3. ドキュメント種別: 仕様書/アーキテクチャ/ADR/手順書/DesignDoc/不具合調査 等 (プロジェクトに種別定義がある場合はそれを選択肢に使う)

### Step 2: 章立ての確定 (Plan mode で実行)

テーマ・目的・想定読者に基づいて章立てを提案し、ユーザーに確認する。

### Step 3: セクションごとに執筆

Principles・プロジェクトの規約に従って各セクションを執筆する。

## Polishing phase

Bootstrap phase で書かれた文章 (新規執筆モード) または既存の文章 (改訂・推敲モード) を入力とし、品質を改善するフェーズ。

最大 3 イテレーション。以下のいずれかで終了する:
- 2 回連続で修正指摘がゼロ
- 3 イテレーション到達

### 各イテレーションの流れ

#### 1. 2 つの レビュー を Subagent として並列実行
##### 1-1. レビュー Agent 1

以下のチェックリストでドキュメントを評価し、指摘事項をリスト形式で報告する。指摘なしの場合は「指摘なし」と報告する。

- [ ] 同じ趣旨の文が繰り返されていないか → 重複排除を提案
- [ ] 1つのパラグラフに複数のトピックが混在していないか → 分割を提案
- [ ] テーマ・目的・想定読者を逸脱した情報がないか → 削除または外部参照化を提案
- [ ] 論拠が不足している主張がないか → 不足箇所を指摘 (情報収集はしない)
- [ ] 表や箇条書きで表現した方が明瞭になる箇所がないか → 変換を提案

##### 1-2. 構造化レビュー

ドキュメントを構造化したうえで再構築することで、不必要な記述を見つけ出す。

###### 1-2-1. レビュー Agent 2-1

ドキュメントを「情報の階層化」「情報の並列化」「情報の比較」「時間軸」「因果関係」の観点で構造化し、簡潔なドキュメントとして再構築する。

###### 1-2-2. レビュー Agent 2-2

元のドキュメントとレビュー Agent 2-1 で再構築されたドキュメントを比較し、元のドキュメントに含まれる本質的ではない記述の省略を提案する。指摘なしの場合は「指摘なし」と報告する。

#### 2. 執筆 Agent (Main session で実行)**

- レビュー Agent の指摘に基づいて修正する
- 修正不要と判断した指摘にはその理由を記録し、次のレビュー Agent の渡す
- プロジェクト規約が存在したら、Validation を行う
  - プロジェクトにドキュメントレビュースキル・Rules がある場合は実行する
  - プロジェクトに lint ツールがある場合は実行する
- すべての指摘を処理したら次のイテレーションへ (または終了)
